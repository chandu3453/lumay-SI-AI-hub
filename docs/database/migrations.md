# Database Migrations

## Tooling

Migrations use [Alembic](https://alembic.sqlalchemy.org/) with async SQLAlchemy support.

## Migration Workflow

### Create a New Migration

```bash
cd backend
alembic revision --autogenerate -m "add_complaint_severity_index"
```

Review the generated file in `backend/alembic/versions/` before applying.

### Apply Migrations

```bash
# Upgrade to latest
alembic upgrade head

# Upgrade to a specific revision
alembic upgrade <revision_id>

# Rollback one step
alembic downgrade -1

# Rollback to a specific revision
alembic downgrade <revision_id>
```

### View Migration History

```bash
alembic history
alembic current
```

## Conventions

1. **One migration per logical change** — do not bundle unrelated schema changes.
2. **Always review autogenerated migrations** — Alembic may miss rename detections or index changes.
3. **Include both `upgrade()` and `downgrade()`** — every migration must be reversible.
4. **Use `batch_mode` for ALTER operations** on tables with data.
5. **Name migrations descriptively**: `add_complaint_category_index`, `create_customer_addresses_table`.

## Model Registration

Domain models are registered in `backend/alembic/env.py`:

```python
from domains.identity.models import User       # noqa: F401
from domains.customer.models import Customer    # noqa: F401
from domains.complaint.models import Complaint  # noqa: F401
# ... all domain models
```

## Testing

```bash
# Test migration in a fresh database
alembic upgrade head
alembic downgrade -1
alembic upgrade head
```

Run migration tests in CI to catch import or ordering issues.
